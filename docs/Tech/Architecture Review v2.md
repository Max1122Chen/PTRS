# 个性化旅游推荐系统架构复盘（v2）

## 1. 需求分析与设计问题复盘

### 1.1 需求分析问题

#### 1.1.1 概念定义不清晰
- **问题**：在需求分析阶段，没有明确校园与景区的关系
- **影响**：导致设计中出现了重复的表结构（campuses和scenic_areas），造成数据冗余和管理混乱
- **原因**：需求文档中对校园和景区的定义不够清晰，没有明确它们之间的包含关系
- **解决方案**：明确校园是景区的一种类型，统一使用scenic_areas表管理，通过type字段区分

#### 1.1.2 道路归属问题
- **问题**：设计中道路（roads）隶属于某个景区（area_id），但实际中大量城市道路不在景区内
- **影响**：导致城市道路无法被正确管理，路线规划功能受限
- **原因**：需求分析时没有考虑城市道路的普遍存在性，默认道路都在景区内
- **解决方案**：移除roads表中的area_id字段，道路可以独立存在，通过起点和终点与其他实体关联

### 1.2 技术选型问题

#### 1.2.1 技术栈过于复杂
- **问题**：作为课程设计项目，使用了Spring Cloud等复杂技术
- **影响**：增加了开发复杂度，超出了课程设计的合理范围
- **原因**：技术选型时追求完整性，没有充分考虑项目规模和开发难度
- **解决方案**：简化技术栈，移除Spring Cloud等复杂技术，只保留必要的核心技术

### 1.3 数据库设计问题

#### 1.3.1 外键约束问题
- **问题**：大量使用外键约束，可能影响查询性能
- **影响**：在数据量大的情况下，外键约束会降低查询和插入性能
- **原因**：设计时遵循了传统数据库设计规范，但没有考虑性能优化
- **解决方案**：移除外键约束，在应用层实现数据完整性校验

## 2. 架构设计优化方案

### 2.1 数据模型优化

#### 2.1.1 景区与校园统一
- **方案**：使用scenic_areas表统一管理景区和校园，通过type字段区分
- **具体措施**：
  - 添加type字段，支持"普通景区"、"校园"等类型
  - 移除单独的campuses表
  - 更新所有关联表，只保留area_id字段

#### 2.1.2 道路独立管理
- **方案**：道路不再隶属于特定景区，成为独立的基础设施
- **具体措施**：
  - 移除roads表中的area_id字段
  - 通过起点和终点ID与建筑物、设施等实体关联
  - 路线规划时根据起点和终点的位置自动计算路径

### 2.2 技术栈简化

#### 2.2.1 后端技术栈
- **保留**：Java 11、Spring Boot 2.7.x、MyBatis-Plus 3.5.x、MySQL 8.0
- **移除**：Spring Cloud、Elasticsearch
- **调整**：Redis作为可选组件，根据实际需求决定是否使用

#### 2.2.2 前端技术栈
- **保留**：Vue 3.x、Vue Router 4.x、Pinia 2.x、Axios 1.x、Element Plus 2.x、Leaflet 1.9.x
- **移除**：ECharts（可选，根据实际需求决定）

### 2.3 数据库设计优化

#### 2.3.1 移除外键约束
- **方案**：移除所有外键约束，在应用层实现数据完整性
- **具体措施**：
  - 移除所有FOREIGN KEY约束
  - 在Service层实现关联校验
  - 添加适当的索引提高查询性能

#### 2.3.2 索引优化
- **方案**：根据查询需求添加合理的索引
- **具体措施**：
  - 为常用查询字段添加索引
  - 优化复合索引结构
  - 避免过度索引

## 3. 实施计划

### 3.1 需求分析阶段
1. **概念梳理**：明确所有核心概念的定义和关系
2. **边界确定**：明确系统的功能边界和数据边界
3. **场景分析**：分析各种使用场景，确保设计的完整性

### 3.2 架构设计阶段
1. **技术选型**：根据项目规模和需求选择合适的技术栈
2. **数据模型设计**：设计清晰、高效的数据模型
3. **接口设计**：设计简洁、易用的API接口

### 3.3 实现阶段
1. **代码实现**：按照设计文档实现功能
2. **测试验证**：确保系统功能正常
3. **性能优化**：优化系统性能

## 4. 结论

本次复盘识别了需求分析和设计中的关键问题，包括概念定义不清晰、技术选型过于复杂、数据库设计不够优化等。通过优化方案的实施，可以构建一个更清晰、更高效、更符合课程设计要求的个性化旅游推荐系统。

架构设计应该始终以需求为导向，根据项目规模和实际需求选择合适的技术方案，避免过度设计和技术复杂化。同时，数据库设计应注重性能优化，在保证数据完整性的前提下，选择更灵活的实现方式。
